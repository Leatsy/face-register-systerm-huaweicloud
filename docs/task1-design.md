# 人脸签到系统设计文档（任务 1）

## 1. 设计目标

根据 `README.md` 中任务 1 的要求，系统需要围绕“手机端拍照上传 + 云端识别 + 结果反馈”的流程完成总体设计，并说明以下问题：

1. 系统整体架构如何组织。
2. 人脸检测与编码模型如何选择。
3. 如何满足注册、登录、照片库管理、签到、签到记录查看等功能。
4. 在云端部署、公网访问、高并发场景下如何扩展。

结合当前项目的实际实现，本系统已经形成一套可落地的方案：前端采用 React 移动端 H5，后端采用 FastAPI，识别服务采用 `InsightFace + onnxruntime`，部署目标为华为云单机服务器，数据库使用华为云 RDS MySQL，图片存储兼容本地文件与华为云 OBS。

## 2. 需求分析

### 2.1 功能需求

根据课程任务描述，系统至少需要提供以下功能：

1. 照片库管理
   - 用户需要登录后才能访问个人信息与标准照片管理功能。
   - 支持注册时上传标准人脸照片。
   - 支持后续更新标准照片，并重新录入人脸特征。
2. 签到
   - 用户无需登录即可执行签到。
   - 用户在手机端上传现场照片，系统在服务端完成人脸检索。
   - 若识别到已录入用户，则生成签到成功记录。
   - 若未识别到用户，则生成失败或未知人员记录。
3. 签到记录查看
   - 登录用户可查看自己的签到历史记录。
4. 账号管理
   - 支持注册、登录、注销。
5. 签到事件管理
   - 已登录用户可以发布签到事件。
   - 系统只展示“已开始且未结束”的事件供签到使用。

### 2.2 非功能需求

1. 支持公网访问，部署在云服务器上。
2. 使用视觉神经网络完成人脸检测与特征编码。
3. 兼容移动端高像素照片上传。
4. 对并发签到请求具备一定扩展预案。
5. 部署方式尽量简单，适合课程演示和单机上线。

## 3. 总体架构

### 3.1 部署约束

当前项目的运行环境与实际约束如下：

1. 本地开发环境为 WSL。
2. 生产环境为一台华为云 Linux 服务器。
3. 结构化数据使用华为云 RDS MySQL。
4. 图片文件支持两种后端：
   - 开发阶段使用服务器本地目录。
   - 生产阶段可切换至华为云 OBS。
5. 后端使用 Docker / Docker Compose 部署。
6. 前端构建产物由 Nginx 托管，并通过 `/api` 反向代理后端接口。

### 3.2 分层架构

系统采用“前端展示层 + API 服务层 + 人脸识别层 + 数据存储层”的结构。

```text
移动端 H5 / 浏览器
  ├─ 首页
  ├─ 登录 / 注册
  ├─ 签到事件列表
  ├─ 拍照或选图签到
  └─ 个人信息 / 签到记录
            |
            v
Nginx
  ├─ 托管前端静态文件
  ├─ 提供 HTTPS
  └─ /api 反向代理到 FastAPI
            |
            v
FastAPI 后端
  ├─ 认证接口
  ├─ 用户接口
  ├─ 签到事件接口
  ├─ 签到记录接口
  ├─ 图片存储服务
  └─ 人脸识别服务
            |
            +--> InsightFace + ONNX Runtime
            |     ├─ 人脸检测
            |     ├─ 关键点/对齐
            |     ├─ 特征提取
            |     └─ 余弦相似度匹配
            |
            +--> RDS MySQL
            |     ├─ users
            |     ├─ face_embeddings
            |     ├─ attendance_events
            |     └─ attendance_records
            |
            +--> 本地 uploads / 华为云 OBS
```

### 3.3 架构选择理由

1. React H5 页面适合快速构建移动端可访问的演示系统，不依赖小程序审核流程。
2. FastAPI 适合文件上传、表单处理、JSON API 输出与快速联调。
3. `InsightFace` 已集成人脸检测与 ArcFace 风格特征提取，工程化程度高。
4. Docker Compose 适合单台云服务器上的低成本部署。
5. Nginx 负责 HTTPS 与同源反代，可以解决浏览器对登录表单和密码传输的安全提示问题。

## 4. 技术栈设计

### 4.1 前端

前端采用 `React + TypeScript + Vite`，实现如下页面：

1. 首页
   - 展示“进入签到”“签到发布”和右上角登录/头像入口。
2. 登录页
   - 用户使用学号/工号和密码登录。
3. 注册页
   - 填写用户名、姓名、手机号、密码，并上传标准人脸照片。
4. 签到事件列表页
   - 按时间筛选当前可签到事件。
5. 签到页
   - 用户上传签到照片并获得签到结果。
6. 个人信息页
   - 展示个人资料、标准照片更新入口和个人签到记录。
7. 签到发布页
   - 发布新的签到事件。

前端另外实现了以下工程性优化：

1. 图片选择后本地预览。
2. 移动端自动压缩大图，减少 `413 Request Entity Too Large` 问题。
3. 对后端返回非 JSON 响应做友好错误提示。
4. 对登录与注册表单补充 `autocomplete` 属性，改善浏览器安全提示和自动填充体验。

### 4.2 后端

后端采用 `FastAPI + SQLAlchemy + Pydantic`：

1. `FastAPI`
   - 提供 REST API。
   - 处理文件上传、鉴权、参数校验。
2. `SQLAlchemy 2.x`
   - 负责 ORM 映射与数据库访问。
3. `PyJWT`
   - 负责 JWT 登录态。
4. `PBKDF2-SHA256`
   - 用于密码哈希存储。
5. `uvicorn`
   - 作为 ASGI 服务运行入口。

### 4.3 数据存储

1. 开发环境
   - 可使用 SQLite 快速联调。
2. 生产环境
   - 使用华为云 RDS MySQL。
3. 图片存储
   - 开发或简化部署使用本地 `uploads` 目录。
   - 生产环境可通过 OBS SDK 切换到华为云 OBS。

### 4.4 部署

1. 本地开发
   - 使用 `uv` 管理 Python 依赖。
2. 生产构建
   - Docker 镜像内部通过 `pip install -r requirements.txt` 安装依赖。
3. 运行方式
   - 使用 `docker compose up -d --build` 启动后端。
4. 反向代理
   - Nginx 托管前端静态文件，并通过 `/api` 与 `/uploads` 代理后端。

## 5. 人脸识别方案设计

### 5.1 模型选择

当前实现已经确定采用 `InsightFace` 方案，默认模型包为 `buffalo_l`。

该模型方案包含：

1. 检测模型
   - 使用 `buffalo_l` 内置的人脸检测能力，可理解为 RetinaFace / SCRFD 风格检测器。
2. 特征编码模型
   - 使用 ArcFace 风格的人脸 embedding。
3. 推理引擎
   - 使用 `onnxruntime` 在 CPU 上执行推理。

### 5.2 选择理由

1. 识别效果明显优于早期基于 OpenCV 的轻量方案。
2. 模型输出为固定维度 embedding，便于数据库存储和相似度检索。
3. 适合单机 CPU 推理，部署门槛低。
4. 能保持现有 FastAPI 接口不变，只替换识别服务内部实现。

### 5.3 识别流程

对一张上传图片，服务端执行以下步骤：

1. 将上传图片解码为 OpenCV 图像。
2. 使用 `InsightFace` 检测人脸。
3. 从多张人脸中选择最优目标人脸。
4. 提取并归一化 embedding。
5. 与数据库中所有已录入的 embedding 计算余弦相似度。
6. 取最高相似度对应用户作为候选匹配结果。
7. 若分数高于阈值，则返回识别成功；否则返回“未识别到已录入用户”。

### 5.4 阈值方案

当前系统使用余弦相似度阈值，默认值为 `0.48`。

设计理由如下：

1. ArcFace 风格 embedding 适合用余弦相似度判定。
2. 在单张标准照底库下，阈值不能过高，否则容易漏识别。
3. 在课堂签到场景中，更需要先保证可用性，再逐步通过样本调整精度。

实际部署时可按如下范围微调：

1. `0.45` 左右：更宽松，识别成功率更高，但误识别风险上升。
2. `0.48` 左右：当前默认平衡值。
3. `0.50 ~ 0.55`：更严格，适合标准照质量较高的场景。

### 5.5 历史数据兼容

为了兼容旧版数据库结构，系统保留 `face_embeddings` 表的 `embedding_json` 和 `model_name` 字段，不新增向量数据库。

同时采用以下兼容策略：

1. 若旧用户 embedding 来自旧模型或旧方案，则在签到时自动重新从其标准照提取 embedding。
2. 重新提取成功后回写数据库。
3. 因此升级到 `InsightFace` 时不必强制清空已有用户数据。

## 6. 模块设计

### 6.1 认证模块

负责注册、登录、注销与鉴权。

1. 注册
   - 接收用户基础信息与标准照片。
   - 生成人脸特征并写入用户底库。
2. 登录
   - 使用学号/工号与密码换取 JWT。
3. 注销
   - 后端返回注销成功提示，前端删除本地 token。

### 6.2 用户与照片模块

负责个人资料查看与标准照片更新。

1. 查看当前登录用户信息。
2. 更新标准照片。
3. 更新成功后重新提取并保存新的人脸 embedding。

### 6.3 签到事件模块

用于管理签到时间窗口。

1. 事件名称、描述、开始时间、结束时间。
2. 开始时间由服务器生成，避免前端时间不可信。
3. 结束时间默认为发布时间后一小时，但允许调整。
4. 前端只展示“已开始且未结束”的事件。

### 6.4 签到模块

负责实际的识别与记录落库。

1. 用户选择签到事件并上传现场照片。
2. 服务端完成人脸识别。
3. 若识别到已录入用户，则创建 `success` 记录。
4. 若未识别到用户，则创建 `unknown` 记录。
5. 对重复签到的用户做幂等处理，避免多次成功写入。

### 6.5 记录查询模块

负责登录用户查看历史签到记录。

1. 返回当前用户的个人签到记录。
2. 记录包含事件信息、时间、消息、匹配分数和照片地址。

## 8. 核心流程设计

### 8.1 注册流程

1. 用户进入注册页，填写学号、姓名、手机号、密码。
2. 用户上传标准人脸照片。
3. 前端先对图片进行压缩，减小上传体积。
4. 后端保存用户基础信息与标准照片。
5. 识别服务提取人脸 embedding。
6. 成功则写入 `face_embeddings` 并完成注册。
7. 前端自动执行登录并跳转首页。

### 8.2 登录流程

1. 用户输入账号密码。
2. 后端校验 `password_hash`。
3. 登录成功后返回 JWT。
4. 前端将 token 保存到本地，并用于访问需要鉴权的接口。

### 8.3 发布签到事件流程

1. 登录用户进入签到发布页。
2. 前端提交事件标题、描述和结束时间。
3. 后端以当前服务器时间作为开始时间。
4. 后端校验结束时间必须晚于开始时间。
5. 写入 `attendance_events`。

### 8.4 签到流程

1. 用户进入签到事件列表，选择一个进行中的事件。
2. 用户上传现场照片。
3. 后端检查事件是否存在、是否已开始、是否已结束。
4. 识别服务从上传图像中提取 embedding。
5. 系统与底库逐个计算余弦相似度。
6. 若最高分超过阈值，则识别为对应用户并生成成功记录。
7. 若未超过阈值，则生成未知人员记录。
8. 前端根据结果显示“签到成功”或“未识别到已录入用户”。

### 8.5 更新标准照片流程

1. 用户登录后进入个人信息页。
2. 上传新的标准人脸照片。
3. 后端重新提取 embedding。
4. 新增一条 `face_embeddings` 记录，并更新头像地址。

### 8.6 查看签到记录流程

1. 用户登录后请求个人签到记录。
2. 后端根据 JWT 识别用户身份。
3. 查询该用户的签到记录并按时间倒序返回。

## 9. 接口设计

当前系统已经实现并对外稳定的核心接口如下。

### 9.1 认证接口

### 9.2 用户接口

1. `GET /api/users/me`
   - 获取当前用户信息。
2. `POST /api/users/me/face-photo`
   - 更新标准照片并重新录入特征。

### 9.3 签到事件接口

1. `GET /api/events`
   - 获取签到事件列表。
2. `POST /api/events`
   - 发布签到事件。

### 9.4 签到接口

1. `POST /api/attendance/check-in`
   - 无需登录，上传签到照片与事件编号。
2. `GET /api/attendance/records`
   - 登录用户查看个人签到记录。

## 10. 高并发处置设计

课程项目当前部署在单台服务器上，但设计上需要说明高并发扩展思路。

### 10.1 当前实现的基本策略

1. 前端先压缩图片，减少网络与后端解码压力。
2. `InsightFace` 模型通过缓存单例加载，避免每次请求重复初始化。
3. Docker Compose 中将模型目录挂载到宿主机，避免容器重建时反复下载模型。
4. 登录与个人信息接口、签到接口分离，降低耦合。

### 10.2 单机阶段的优化方向

1. 使用 Nginx 做静态资源分发与接口反向代理。
2. 开启 `client_max_body_size` 控制上传上限。
3. 增加 API 层限流，抑制恶意重复签到。
4. 将历史 embedding 预热到内存，减少每次请求对数据库的读取。

### 10.3 扩展阶段的设计预案

当用户规模或签到并发继续提升时，可按以下路径扩展：

1. 使用 Redis 缓存用户 embedding 元数据与热点事件。
2. 将签到识别改造成“上传后入队，后台 worker 推理”的异步模式。
3. 使用 Celery / RQ 等任务队列拆分 API 与推理服务。
4. 将向量检索从线性扫描升级为 `FAISS` 或向量数据库。
5. 将 Web API 与推理服务拆成两个容器或两台机器。

### 10.4 高并发瓶颈分析

当前最主要瓶颈不是数据库，而是图像解码与模型推理。

因此系统扩展优先级应为：

1. 压缩上传图片。
2. 缓存模型与底库。
3. 异步化识别请求。
4. 向量检索优化。

## 11. 安全设计

### 11.1 账号安全

1. 密码采用哈希存储，不保存明文。
2. 登录后使用 JWT 进行鉴权。
3. 仅对用户个人资料与签到记录接口要求登录。

### 11.2 传输安全

1. 前后端生产环境应通过 HTTPS 部署。
2. Nginx 采用同源 `/api` 反代，避免浏览器对登录表单出现“信息泄露风险”提示。
3. 建议启用 `Strict-Transport-Security`。

### 11.3 上传安全

1. 限制上传文件类型为图片。
2. 前端先压缩图片，后端再做大小控制。
3. 对识别失败和异常情况返回明确错误消息。

### 11.4 存储安全

1. RDS 仅允许应用服务器访问。
2. OBS 或本地上传目录应避免直接暴露敏感写权限。
3. 保留签到现场照片和分数，便于后续审计与复核。

## 12. 部署设计

### 12.1 单机部署方案

当前推荐部署方式为单台华为云服务器同时承载：

1. Nginx
2. 前端静态文件
3. FastAPI Docker 容器

数据库放在华为云 RDS，图片可放本地或 OBS。

### 12.2 后端部署

1. 后端镜像基于 `python:3.12-slim`。
2. 依赖通过 `requirements.txt` 安装。
3. 默认使用华为云 PyPI 镜像源，减轻海外下载问题。
4. `InsightFace` 模型目录挂载为 `./.insightface:/app/.insightface`。

### 12.3 前端部署

1. 前端通过 `npm run build` 生成 `dist`。
2. Nginx 将 `dist` 作为静态目录。
3. 前端生产环境使用 `VITE_API_BASE_URL=/api`，通过同源访问后端接口。

### 12.4 模型部署注意事项

`InsightFace` 模型首次下载可能较慢，因此建议：

1. 本地先完成模型下载。
2. 将 `backend/.insightface` 目录同步到服务器。
3. 容器直接复用服务器上的模型缓存，避免启动时从 GitHub 慢速下载。

## 13. 风险与应对

1. 风险：手机拍照图片过大，上传失败。
   - 应对：前端本地自动压缩，后端限制上传大小。
2. 风险：浏览器提示账号信息泄露风险。
   - 应对：统一改为 HTTPS，同源 `/api` 反代。
3. 风险：模型首次下载过慢。
   - 应对：提前在本地下载并同步到服务器。
4. 风险：单张标准照识别不稳定。
   - 应对：调低阈值、更新标准照、后续支持多标准照。
5. 风险：并发签到时响应慢。
   - 应对：缓存 embedding、异步识别、向量索引化。
