# 人脸签到系统设计文档（任务 1）

## 1. 项目目标

本项目需要设计并实现一个面向课堂/实验场景的人脸签到系统，支持手机端拍照上传、云端服务器识别、签到结果反馈，并逐步完成以下三阶段目标：

1. 任务 1：提交系统设计文档，明确总体架构、技术路线、模型选择与高并发处置思路。
2. 任务 2：完成手机端与服务器端连通的基础框架，演示注册、登录、注销、标准人脸照片上传、拍照上传等流程。
3. 任务 3：完成完整签到功能与签到记录查看功能，演示已录入人员签到、未录入人员签到两种结果。

## 2. 需求分析

### 2.1 功能需求

根据 `README.md`，系统至少需要支持以下能力：

1. 照片库管理
   - 需要登录后访问。
   - 支持注册时录入基本信息并上传标准人脸照片。
   - 支持查看、更新用户照片库。
2. 签到
   - 不需要登录即可签到。
   - 手机端拍照上传签到照片。
   - 服务器端通过人脸检索找到对应用户并完成签到标记。
   - 对未录入人员给出失败反馈。
3. 签到记录查看
   - 需要登录后访问。
   - 可按时间、人员查看签到记录。
4. 基础账号能力
   - 注册。
   - 登录。
   - 注销。

### 2.2 非功能需求

1. 云端部署，支持公网 IP 访问。
2. 使用视觉神经网络完成检测与编码。
3. 需要考虑高并发场景的处理能力。
4. 具备可演示性，能清晰展示前后端连通、页面效果和数据库变化。

## 3. 总体方案

### 3.0 部署环境约束

根据当前已知条件，项目采用如下环境方案：

1. 本地开发环境为 WSL，便于前后端联调与快速迭代。
2. 生产环境部署在一台华为云 Linux 服务器上。
3. 数据库使用华为云 RDS MySQL，不在应用服务器本机自建数据库。
4. 图片文件优先存储到华为云 OBS；本地开发阶段可先落盘到服务器目录。
5. 后端服务通过 Docker 部署，Python 依赖由 `uv` 管理。

### 3.1 架构设计

系统采用“手机端 + 云端 API 服务 + 人脸识别服务 + 数据库/对象存储”的分层架构。

```text
手机端
  ├─ 注册/登录/注销
  ├─ 标准照片上传
  ├─ 拍照签到
  └─ 签到记录查看
        |
        v
API 网关 / Web 服务层
  ├─ 用户认证
  ├─ 用户信息管理
  ├─ 照片上传管理
  ├─ 签到请求接收
  └─ 记录查询接口
        |
        +--> 人脸识别服务
        |     ├─ 人脸检测
        |     ├─ 人脸对齐
        |     ├─ 特征编码
        |     └─ 人脸检索 / 相似度匹配
        |
        +--> 华为云 RDS MySQL
        |     ├─ 用户表
        |     ├─ 人脸特征表
        |     └─ 签到记录表
        |
        +--> 华为云 OBS / 本地文件存储
              ├─ 标准照片
              └─ 签到抓拍照片
```

### 3.2 推荐技术栈

为了兼顾实现效率、课程展示效果和部署难度，建议采用以下技术栈：

1. 手机端：微信小程序或 H5 页面
   - 优先推荐微信小程序，原因是调用摄像头、上传图片和演示流程更方便。
   - 如果团队已有 Web 开发基础，也可以采用 Vue/React 移动端页面。
2. 后端：Python `FastAPI`
   - 适合快速构建 REST API。
   - 原生支持异步接口、文件上传、自动生成接口文档。
   - 与 Python 视觉模型生态兼容性高。
3. 人脸识别：`InsightFace` 或 `facenet-pytorch`
   - 检测与特征提取一体化能力较强，工程化方便。
4. 数据库：`MySQL`
   - 结构化数据管理简单，便于截图展示数据变化。
   - 生产环境直接使用华为云 RDS MySQL。
5. 缓存与削峰：`Redis`
   - 用于登录态缓存、任务排队、热点数据缓存。
6. 文件存储：本地磁盘或对象存储
   - 本地开发阶段可先用服务器本地目录。
   - 生产环境推荐使用华为云 OBS 保存图片。
7. 部署：华为云 Linux 服务器 + Docker + `Nginx` + `uvicorn`
8. Python 依赖管理：`uv`
   - 本地开发和容器构建均基于 `pyproject.toml` 管理依赖。

## 4. 模型选择

### 4.1 检测模型

推荐使用 `SCRFD` 或 `RetinaFace` 进行人脸检测。

选择理由：

1. 检测精度较高，适合课堂拍照和常见光照场景。
2. 对多人脸场景有较好的鲁棒性。
3. 与常见人脸特征提取模型兼容性较好。

### 4.2 特征编码模型

推荐使用 `ArcFace` 系列模型进行人脸特征编码。

选择理由：

1. ArcFace 是成熟的人脸识别方案，公开资料丰富。
2. 生成的 embedding 向量便于做相似度检索。
3. 与 `InsightFace` 工具链配合时，检测、对齐、编码流程较完整。

### 4.3 识别策略

服务器端不直接依赖设备绑定，而是采用“人脸向量检索 + 阈值判断”的方式：

1. 对签到照片做人脸检测。
2. 选取置信度最高的人脸或要求单人脸输入。
3. 生成人脸 embedding。
4. 与已注册用户的人脸特征库计算余弦相似度。
5. 相似度超过阈值则判定为对应用户签到成功。
6. 低于阈值则判定为未录入人员或识别失败。

### 4.4 阈值方案

初始方案可采用经验阈值，例如：

1. 单张标准照注册时，阈值设为 `0.40 ~ 0.50` 的余弦距离上限，或等价相似度下限。
2. 后续通过实际采样数据微调阈值，减少误识别和漏识别。

说明：不同库对“距离/相似度”的定义不同，实际实现时需根据所选模型输出统一标准。

## 7. 核心流程设计

### 7.1 注册流程

1. 用户在手机端填写基本信息。
2. 上传标准人脸照片。
3. 后端校验参数并保存用户信息。
4. 人脸识别服务检测照片中是否存在且仅存在一张清晰人脸。
5. 生成人脸特征向量并写入 `face_embeddings`。
6. 返回注册成功结果。

### 7.2 登录流程

1. 用户输入账号密码。
2. 服务端校验密码哈希。
3. 返回 JWT 或会话令牌。
4. 手机端在后续需要鉴权的接口中携带 Token。

### 7.3 签到流程

1. 用户在签到页拍照上传。
2. 服务端接收照片，保存原图。
3. 人脸服务检测人脸并提取特征。
4. 与人脸库做相似度匹配。
5. 若匹配成功：
   - 生成签到记录。
   - 返回姓名、学号、签到成功时间。
6. 若匹配失败：
   - 生成失败记录。
   - 返回“未录入人员”或“识别失败”提示。

### 7.4 记录查看流程

1. 登录用户携带 Token 请求记录接口。
2. 服务端校验权限。
3. 返回个人或管理员可查看的签到记录列表。

## 8. 接口设计建议

### 8.1 认证接口

1. `POST /api/auth/register`
   - 注册并上传标准人脸照片。
2. `POST /api/auth/login`
   - 登录。
3. `POST /api/auth/logout`
   - 注销。

### 8.2 用户与照片接口

1. `GET /api/users/me`
   - 获取当前用户信息。
2. `POST /api/users/me/face-photo`
   - 更新标准人脸照片。
3. `GET /api/users/me/face-photo`
   - 查看当前照片信息。

### 8.3 签到接口

1. `POST /api/attendance/check-in`
   - 无需登录，上传签到照片并返回匹配结果。

### 8.4 记录接口

1. `GET /api/attendance/records`
   - 查看签到记录。

## 9. 高并发处置设计

虽然课程演示阶段用户量可能不大，但任务要求明确提出“考虑高并发量处置”，因此设计上需要预留扩展能力。

### 9.1 请求分层

1. `Nginx` 负责静态资源、反向代理和基础限流。
2. Docker 容器内运行 `FastAPI` 服务，负责轻量 API 处理。
3. 耗时的人脸识别任务可异步化，交给后台 worker 处理。

### 9.2 缓存与削峰

1. 将已注册用户的人脸向量加载到内存或 Redis 中，避免每次都从数据库读取。
2. 对短时间高频签到请求进行限流，防止同一设备重复提交。
3. 高峰时通过任务队列异步处理图片，前端轮询结果。

### 9.3 索引优化

1. 当前用户量较小时，可直接做 embedding 线性扫描。
2. 用户量增大后，接入 `FAISS` 做向量索引，提高检索速度。
3. 数据库对学号、时间字段建立索引，优化记录查询性能。

### 9.4 资源隔离

1. Web API 与模型推理可拆分为两个进程或两个服务。
2. 图片上传先落盘，再异步做人脸分析，降低接口阻塞时间。
3. 通过日志和监控观察响应时间、失败率、队列长度。

## 10. 安全与可靠性设计

1. 密码必须做哈希存储，不能明文保存。
2. 上传图片需限制格式、大小与访问权限。
3. 注册照片要求单人脸、无遮挡、光照正常，降低误识别率。
4. 签到接口需记录原图、匹配分数和结果，便于复核。
5. 对敏感接口加入身份验证和基本限流。
6. 数据库定期备份，防止演示数据丢失。
7. 华为云环境下应通过安全组仅开放必要端口，并限制 RDS 访问白名单。

## 12. 风险与应对

1. 风险：光照、姿态、遮挡导致识别失败。
   - 应对：注册阶段要求上传标准正脸照片，签到时提示对准镜头。
2. 风险：多人同时签到导致响应慢。
   - 应对：增加缓存、异步队列、向量索引。
3. 风险：误识别。
   - 应对：设置合理阈值，记录匹配分数，必要时采用多张底库照片。
4. 风险：演示环境不稳定。
   - 应对：提前部署到云服务器，保留本地备份环境。

## 13. 结论

本方案采用“手机端采集 + 云端 API + 人脸识别服务 + 数据库存储”的整体架构，能够满足课程要求中的注册、登录、照片库管理、无登录签到、签到记录查看等功能，并通过 ArcFace 等成熟方案实现人脸检测与编码。设计上已经预留缓存、异步任务、向量检索等机制，可支撑后续高并发扩展。该文档可作为任务 1 的提交材料，也可直接作为任务 2、任务 3 的开发蓝图。
